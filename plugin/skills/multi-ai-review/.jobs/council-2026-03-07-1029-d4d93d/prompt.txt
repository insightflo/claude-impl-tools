# Review Request: AI 에이전트 팀에 인간 팀 구조를 적용하는 것이 비효율적인가?

## 배경

project-team이라는 AI 에이전트 시스템이 있다. 현재 9개의 에이전트가 인간 팀을 모방하여 구성되어 있다:

```
현재 에이전트 (인간 팀 모방):
- ProjectManager.md    (프로젝트 매니저)
- ChiefArchitect.md    (수석 아키텍트)
- BackendSpecialist.md (백엔드 개발자)
- FrontendSpecialist.md (프론트엔드 개발자)
- QAManager.md         (QA 매니저)
- MaintenanceAnalyst.md (유지보수 분석가)
- ChiefDesigner.md     (수석 디자이너)
- DBA.md               (데이터베이스 관리자)
- SecuritySpecialist.md (보안 전문가)
```

이전 Council 리뷰에서는 이를 3개 핵심 에이전트(Lead, Builder, Reviewer)로 통합하고, 전문가(DBA, Security, Designer)는 필요시 소환하는 방식을 권고했다.

## 핵심 질문

**사용자의 원래 의도**: 인간 소프트웨어 개발팀의 역할 구분을 AI 에이전트에 그대로 반영하여, 각 역할이 전문성을 갖고 협업하는 구조를 만들고 싶었다.

이에 대해 다음을 분석해 달라:

### 1. 인간 팀 구조 모방의 장점
- 역할 전문화에 의한 품질 향상 가능성
- 검증과 균형(checks & balances) 메커니즘
- 도메인 지식 집중
- 인간이 이해하기 쉬운 조직 구조

### 2. AI 에이전트에서 인간 팀 구조의 한계
- LLM은 이미 멀티도메인 역량을 내재하고 있음 (같은 모델이 PM도 되고 개발자도 됨)
- 에이전트 간 통신 오버헤드 (컨텍스트 전달 비용)
- 컨텍스트 윈도우 분산에 의한 정보 손실
- 에이전트 수가 많을수록 조율 복잡성 증가

### 3. "비효율적"이라고 단정할 수 있는가?
- 프로젝트 규모에 따른 적정 에이전트 수
- 역할 전문화 vs 컨텍스트 공유의 trade-off
- "인간 팀과 같은 구성"이라는 아이디어 자체의 가치
- 어떤 상황에서는 인간 팀 구조가 AI에서도 유효할 수 있는가?

### 4. 절충안의 타당성
이전 Council이 제안한 "3 core + specialist on-demand" 구조가:
- 인간 팀 모방의 장점을 살리면서
- AI 특성에 맞게 최적화한 것인지
- 아니면 다른 더 나은 접근이 있는지

## 컨텍스트: Claude Code 에이전트 시스템

이 에이전트들은 Claude Code의 서브에이전트(Agent tool)로 동작한다:
- 각 에이전트는 `.md` 파일로 정의 (frontmatter + 프롬프트)
- 도구 접근 권한을 frontmatter에서 제어 (Read, Write, Edit, Bash 등)
- 17개의 deterministic hooks가 규칙 강제
- Lead → Builder → Reviewer 순으로 위임/실행/검증
- 에이전트 수가 많을수록 설치/관리 복잡성 증가

## 평가 기준

1. **효율성**: 토큰 소비, 실행 시간, 컨텍스트 활용률
2. **품질**: 코드 품질, 아키텍처 일관성, 버그 발견률
3. **실용성**: 설치 편의성, 학습 곡선, 유지보수 부담
4. **확장성**: 프로젝트 규모별 적응력

자유롭게 의견을 제시해 달라. "비효율적이다"가 맞는지, "상황에 따라 다르다"가 맞는지, "오히려 효율적이다"가 맞는지.
