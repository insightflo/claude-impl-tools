## 시나리오 1: 1인 개발자 - SaaS MVP 고속 착수

### 배경
아이디어만 있는 1인 프리랜서 개발자가 2주 안에 MVP를 완성하려 함. 메모장에 적어둔 기존 할 일 목록이 있고, 프로젝트 구조부터 핵심 기능까지 빠르게 진행하고 싶어서 Claude에게 일임.

### 사용자 입력
```
뭐부터 시작해야 해?

# 기존 메모/레거시 태스크가 있는 경우
/tasks-migrate

# 새로 시작하는 경우
/tasks-init
/agile auto
```

### 호출 흐름

#### 0단계: 워크플로우 가이드
- **트리거 스킬**: `workflow-guide` ("뭐해야해?", "어떤 스킬?", `/workflow`)
- **SKILL.md 위치**: `skills/workflow-guide/SKILL.md`
- **동작**: 현재 상태(TASKS.md 유무, 태스크 수, 에이전트 수)를 bash로 silent 분석 → 상황별 추천 스킬 안내 (스크립트 없음, Claude 직접 수행)

#### 1단계: 기존 태스크 마이그레이션 (선택)
- **트리거 스킬**: `tasks-migrate` (`/tasks-migrate`)
- **SKILL.md 위치**: `skills/tasks-migrate/SKILL.md`
- **동작**: 레거시 태스크 파일(`docs/planning/06-tasks.md`, TODO 주석 등) → 루트 `TASKS.md`로 통합 (스크립트 없음, Claude 직접 수행)
- **파일 R/W**:
  - **읽기**: 기존 태스크 파일들, TODO 주석
  - **쓰기**: `TASKS.md` (통합)

#### 2단계: 태스크 초기화
- **트리거 스킬**: `tasks-init` (`/tasks-init`)
- **실행 스크립트**:
  ```
  skills/tasks-init/scripts/tasks-init.sh   # 진입점, 환경 확인
  skills/tasks-init/scripts/analyze.js      # 프로젝트 구조/요구사항 분석
  skills/tasks-init/scripts/generate.js     # TASKS.md 스캐폴딩 생성
  ```
- **에이전트**: 없음 (Claude 직접 실행)
- **파일 R/W**:
  - **읽기**: `package.json`, `pyproject.toml`, `requirements.txt`, 소스 트리
  - **쓰기**: `TASKS.md` (Skeleton→Muscles→Skin 레이어 태스크 생성, 약 15~25개)
  - **템플릿 읽기**: `skills/tasks-init/templates/TASKS.md`, `templates/task-metadata.yaml`

#### 3단계: Agile 자동 실행
- **트리거 스킬**: `agile` (`/agile auto`, `/sprint`, "스프린트 시작")
- **SKILL.md 위치**: `skills/agile/SKILL.md` (스크립트 없음 — SKILL.md 프롬프트 기반으로 Claude가 직접 수행)
- **동작**: `TASKS.md`를 Skeleton→Muscles→Skin 레이어별로 읽고, 각 태스크를 Claude가 순차 구현. 레이어 완료 시 사용자 체크인
- **에이전트**: Claude 단일 에이전트 (PM + Developer 역할 겸임)
- **파일 R/W**:
  - **읽기**: `TASKS.md`, `.claude/settings.json`
  - **쓰기**: 소스 코드(`src/`), `TASKS.md` 체크박스 업데이트

#### 4단계: 상태바 표시 (자동)
- **트리거**: `statusline` (설치 후 자동 활성화)
- **실행 스크립트**:
  ```
  skills/statusline/hooks/tasks-status-writer.js   # TASKS.md 진행률 계산
  skills/statusline/statusline-segment.sh          # 상태바 Line 3 출력
  ```
- **파일 읽기**: `TASKS.md`

### 기대 출력
```
✅ TASKS.md 생성 완료 (Skeleton 8개 / Muscles 10개 / Skin 7개)
🚀 /agile auto 시작 → Skeleton 레이어부터 구현...
📊 [▓▓▓▓░░░░░░] 4/25 완료
```

**커버 스킬**: `workflow-guide`, `tasks-migrate`, `tasks-init`, `agile`, `statusline`

```

/orchestrate --mode=wave
/task-board show
```

### 호출 흐름

#### 1단계: 거버넌스 팀 설정
- **트리거 스킬**: `governance-setup` (`/governance`, "거버넌스 설정")
- **SKILL.md 위치**: `skills/governance-setup/SKILL.md` (스크립트 없음 — Claude가 직접 수행)

#### 2단계: Project Team 설치
- **스킬 안내에 따라 터미널에서 실행**:

/impact src/domains/payment/refundService.ts

  project-team/hooks/pre-edit-impact-check.js   # 수정 시도 전 자동 실행

  - 옵션: `/deps --cycles`, `/deps <파일> --tree`, `/deps --matrix`

## 시나리오 4: Multi-AI 컨센서스 코드 리뷰

### 배경
중요한 인증 모듈 PR이 올라왔는데, 보안/품질 측면에서 Claude 혼자의 판단이 아닌 Gemini + Codex의 교차 검토까지 받고 싶은 시니어 개발자.

### 사용자 입력
```
이번 PR 인증 모듈 심층 리뷰해줘. Council 소집.
```

### 호출 흐름

#### 3-Stage Pipeline

**Stage 1: Initial Opinions (병렬)**
- **트리거 스킬**: `multi-ai-review` ("리뷰해줘", "council 소집", "여러 AI 의견 들어보자")
- **실행 스크립트**:
  ```
  skills/multi-ai-review/scripts/council.sh          # 메인 진입점, Job 생성
  skills/multi-ai-review/scripts/council-job.sh      # Job runner → council-job.js 위임
  skills/multi-ai-review/scripts/council-job.js      # Job 오케스트레이터 (config 파싱, 멤버 디스패치)
  skills/multi-ai-review/scripts/council-job-worker.js  # 멤버별 워커 (병렬 실행)
  ```
- **설정 파일**: `skills/multi-ai-review/council.config.yaml`
  ```yaml
  members:
    - name: gemini
      command: "gemini --model gemini-3-flash-preview --output-format text -p"
    - name: codex
      command: "codex exec"
  ```
- **동작**:
  - `council.sh` → Job 디렉토리 생성 (`skills/multi-ai-review/scripts/.jobs/council-{timestamp}-{hash}/`)
  - `council-job-worker.js`가 각 멤버 CLI에 동시 전송
  - 정리: `cleanup.sh` (1시간 이상 된 고아 Job 자동 삭제)

**Stage 2: Cross-Review (반박)**
- Gemini가 Codex 의견을 검토 → 반박/동의

## 시나리오 7: 역할별 Multi-AI 모델 라우팅 실행

### 배경
대형 기능 개발 시 프론트엔드는 Gemini(UI 창의성), 백엔드는 Codex(로직 정확성)에 각각 맡기고, 아키텍처 결정은 Claude가 담당하도록 역할별 모델을 분리하고 싶음.

### 사용자 입력
```
# TASKS.md에 태그 설정
- [ ] T2.1: Dashboard UI 구현 [model:gemini]
- [ ] T2.2: Payment API 구현 [model:codex]
- [ ] T2.3: DB 스키마 설계 [owner:dba]

/orchestrate-standalone
```

### 호출 흐름

#### 라우팅 결정
- **트리거 스킬**: `orchestrate-standalone` → 내부 `multi-ai-run` 라우팅
- **라우팅 우선순위**:
  1. 태스크 제목의 `[model:xxx]` 태그 (최우선)
  2. `skills/multi-ai-run/routing.config.yaml`의 `role_routing` (역할명 매핑)
  3. 같은 파일의 `task_type_routing` (태스크 유형)
  4. `.claude/model-routing.yaml` (프로젝트 오버라이드)
  5. `default: claude`

- **실행 스크립트**:
  ```
  skills/orchestrate-standalone/scripts/scheduler.js
    └─ task.model = "gemini" 감지 → worker.js에 전달

  skills/orchestrate-standalone/scripts/worker.js
    ├─ task.model = "gemini" → gemini --model ... -p "{prompt}"
    ├─ task.model = "codex"  → codex exec --sandbox workspace-write "{prompt}"
    └─ task.owner = "dba"    → routing.config.yaml 조회 → claude (기본)
  ```

- **설정 파일**:
  ```yaml
  # skills/multi-ai-run/routing.config.yaml
  cli_models:
    codex:
      args: "--sandbox workspace-write"  # 프로젝트 폴더 직접 R/W
    gemini:
      args: "--output-format text"
  role_routing:
    backend-specialist: codex
    frontend-specialist: gemini
    dba: claude
  ```

- **에이전트 개입**:

  | 태스크 | 모델 | 실제 CLI | 파일 쓰기 방식 |
  |-------|------|---------|-------------|
  | T2.1 Dashboard UI | Gemini | `gemini --model ... -p "..."` | Claude가 출력 받아 Edit 도구로 적용 |
  | T2.2 Payment API | Codex | `codex exec --sandbox workspace-write "..."` | Codex가 직접 파일 수정 |
  | T2.3 DB 스키마 | Claude | Claude 직접 처리 | Edit/Write 도구 |

- **파일 R/W**:
  - **읽기**: `TASKS.md`, `skills/multi-ai-run/routing.config.yaml`, `.claude/model-routing.yaml` (프로젝트 오버라이드 있으면)
  - **쓰기**: 각 에이전트 담당 소스 파일

### 기대 출력
```
[T2.1] Gemini → Dashboard.tsx 생성 완료
[T2.2] Codex → payment-service.ts 생성 완료 (직접 파일 쓰기)
[T2.3] Claude → schema.prisma 업데이트 완료
```

**커버 스킬**: `multi-ai-run`, `orchestrate-standalone` (라우팅)
